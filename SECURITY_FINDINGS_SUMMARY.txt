🔴 CRITICAL ISSUE: Account Lockout Bypass
────────────────────────────────────────────

LOCATION: /backend/accounts/views.py (Lines 44-170, 280-503, 1774-1870, 2051-2120)

PROBLEM:
  The three 2FA methods (TOTP, Backup Codes, Email OTP) use separate lockout
  counters. An attacker can bypass the 15-minute lockout by switching methods:
  
  ✗ Fail TOTP 3 times          → Locked (key: totp_locked_alice)
  ✓ Switch to Backup Code      → NOT locked (different key!)
  ✓ Fail Backup Code 3 times   → Locked (key: backup_code_locked_1)  
  ✓ Switch to Email OTP        → NOT locked (different key!)
  
  Total: 9 attempts instead of 3 ❌

ATTACK SCENARIO:
  User "alice" attempts login:
  1. Try TOTP 3 times (fail)
  2. Try Backup Code 3 times (fail) 
  3. Try Email OTP 3 times (fail)
  4. Total attempts: 9 (increases brute force success rate 3x)

IMPACT: 🔴 CRITICAL
  - Brute force attack success rate increased 300%
  - 15-minute lockout becomes ineffective
  - Violates NIST 800-63B authentication guidelines

FIX SUMMARY:
  Replace method-specific lockout keys with unified account-level key:
  
  ❌ OLD (separate keys per method):
     - totp_locked_alice
     - backup_code_locked_1
     - email_otp_locked_1
  
  ✅ NEW (unified account key):
     - account_2fa_locked_1
     - 2fa_fail_count_1

IMPLEMENTATION:
  1. Add _account_2fa_lock_key() and _record_shared_2fa_failure() functions
  2. Update VerifyTOTPView to check unified lockout
  3. Update VerifyBackupCodeView to check unified lockout
  4. Update VerifyEmailOTPView to check unified lockout
  5. Test: 3 TOTP failures must block backup code (return 429)

FILES TO MODIFY:
  - /backend/accounts/views.py (add functions + update 3 views)

ESTIMATED FIX TIME: 30 minutes
TESTING TIME: 15 minutes
DEPLOYMENT: No downtime required (Redis-only change)

🟠 HIGH ISSUE: Cache TTL Compatibility
─────────────────────────────────────────

LOCATION: /backend/accounts/views.py (Lines 54, 98, 141, 184)

PROBLEM:
  Code uses cache.ttl() which only exists in Django 4.0+
  If deployed on Django < 4.0, raises AttributeError at runtime

IMPACT: 🟠 HIGH
  - Lockout system could fail silently
  - Users might not be properly locked out

FIX:
  Add fallback for older Django versions or use timestamp-based TTL calculation

🟡 MEDIUM ISSUES
─────────────────

ISSUE #1: TOTP Window Inconsistency
  - Registration: valid_window=1 (±1 additional window = 90 seconds)
  - Login: valid_window=0 (exactly current window = 30 seconds)
  
  FIX: Standardize to valid_window=1 everywhere

ISSUE #2: Missing Rate Limiting on Email OTP Send
  - SendEmailOTPView has no throttle_classes
  - Attacker can send unlimited OTPs to any email
  
  FIX: Add EmailOTPThrottle (max 5 sends per day per user)

ISSUE #3: XSS in Dashboard Icon Rendering
  - File: /frontend/src/pages/Dashboard.tsx
  - Uses dangerouslySetInnerHTML without sanitization
  
  FIX: Use DOMPurify to sanitize icon HTML
